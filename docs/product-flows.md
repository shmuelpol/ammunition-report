## 1. עקרונות יסוד

### 1.1 קובץ Excel כמקור אמת יחיד
- **כל הנתונים נשמרים אך ורק בקבצי Excel (.xlsx) ב-Google Drive**.
- לא יהיו קבצי JSON, CSV, או מסד נתונים מקומי כמקור נתונים.
- הסיבה: משתמשים כבר מנהלים נתונים ב-Excel ידנית. אם האפליקציה תשמור בפורמט אחר, ייווצר פער בין "גרסת האפליקציה" ל"גרסת האקסל" — וזה בטוח יישבר.
- משתמש שרוצה לערוך ידנית ב-Drive יוכל לעשות זאת — האפליקציה תקרא את השינויים בטעינה הבאה.

### 1.2 אל תיגע בקבצים שלא יצרת
- האפליקציה **לעולם לא תערוך קובץ שלא נוצר על ידה**.
- קובץ שנוצר ידנית ע"י המשתמש? האפליקציה תקרא ממנו אבל תיצור קובץ חדש משלה.
- זיהוי: קבצים שנוצרו ע"י האפליקציה יכילו **מטא-דאטא מזהה** בשם הגיליון או בתא מוסתר (ראה סעיף 4).

### 1.3 הרשאות מינימליות מ-Google
- האפליקציה תבקש **רק** הרשאה לקבצים שהיא יצרה, או לתיקייה ספציפית שהמשתמש בחר.
- אין גישה לכל ה-Drive. אין קריאת כל הקבצים.

## 2. הרשאות Google Drive — גישה מינימלית

### 2.1 מצב נוכחי — הבעיה
כרגע האפליקציה מבקשת `drive.file` + `drive.readonly`:
- `drive.readonly` = גישת קריאה **לכל** ה-Drive → הרשאה רחבה מדי ← גוגל ידרוש אימות OAuth מתקדם (production verification) כדי לתת אישור ← ביורוקרטיה.
- בלי `drive.readonly`, אי אפשר לחפש תיקיות שהמשתמש יצר ידנית.

### 2.2 הפתרון — שתי אפשרויות

#### אפשרות א': Scope מינימלי `drive.file` בלבד (מומלץ)
| Scope | מה נותן |
|-------|---------|
| `drive.file` | קריאה/כתיבה רק לקבצים שהאפליקציה יצרה |

**איך זה עובד:**
- האפליקציה יכולה **ליצור** תיקייה ולנהל אותה.
- האפליקציה **לא יכולה** למצוא תיקייה שהמשתמש יצר ידנית.
- פתרון: **האפליקציה תיצור את התיקייה בעצמה**, או שהמשתמש ידביק קישור/מזהה של תיקייה קיימת.

**זרימה:**

```
    כניסה ראשונה:
    ┌──────────────────────────────────────────┐
    │  התחברות ל-Google (drive.file scope)     │
    │                                          │
    │  "לאיזו תיקייה לשמור את הנתונים?"       │
    │                                          │
    │  [א] צור תיקייה חדשה "ammunition-report" │
    │  [ב] הדבק קישור לתיקייה קיימת ב-Drive   │
    │                                          │
    └──────────────────────────────────────────┘
    
    כניסות הבאות:
    - התיקייה שמורה ב-localStorage
    - חיבור אוטומטי (אם הטוקן עדיין בתוקף)
```

**יתרונות:**
- ✅ לא צריך OAuth production verification מגוגל
- ✅ Test mode מספיק (עד 100 משתמשי בדיקה, מושלם לשלב נוכחי)
- ✅ הרשאה מינימלית — המשתמש מרגיש בטוח

**מגבלות:**
- ⚠️ קובץ שמשתמש יצר ידנית ב-Drive לא נגיש ← צריך להעלות דרך האפליקציה או להדביק קישור
- ⚠️ קריאת קובץ Excel ישן (שנוצר ידנית) → דורש שהמשתמש יעלה לתיקייה של האפליקציה או **ישתמש ב-Picker API**

#### אפשרות ב': Google Picker API (חלופה לעתיד)
- Picker API מאפשר למשתמש לבחור קובץ/תיקייה מ-Drive **בלי** שהאפליקציה תצטרך scope רחב.
- הוא UI של גוגל שנפתח, המשתמש בוחר, ו-API מחזיר את ה-ID.
- זה **לא** דורש scope נוסף מעבר ל-`drive.file`.
- שווה להוסיף בעתיד למקרה של "ייבא קובץ קיים".

### 2.3 הגדרת תיקייה מותאמת אישית

המשתמש יוכל להגדיר שם תיקייה שונה מ-`ammunition-report` **לפני ההתחברות**:

```
    ┌─────────────────────────────────────────┐
    │  שם תיקייה ב-Drive:                     │
    │  [  ammunition-report  ]  (ברירת מחדל)  │
    │                                          │
    │  --- או ---                               │
    │  הדבק מזהה תיקייה קיימת:                │
    │  [                                    ]  │
    │                                          │
    │  [התחבר ל-Google]                        │
    └─────────────────────────────────────────┘
```

ההגדרה נשמרת ב-`localStorage` ומשמשת בכל הפעלה עתידית.

## 3. מבנה קבצים ב-Drive

```
ammunition-report/                          ← תיקייה שהאפליקציה יצרה (או שהמשתמש הפנה אליה)
│
├── ammunition-types.xlsx                   ← תבנית סוגי תחמושת (template)
│                                              נוצר ע"י האפל' בהתקנה ראשונה, ניתן לעריכה ידנית
│
├── 9294/                                   ← תיקיית יחידה (שם = מספר גדוד/יחידה)
│   ├── 9294-data.xlsx                      ← קובץ הנתונים המרכזי (הפורמט המומלץ)
│   ├── 9294-data.xlsx.backup-20260310T1430 ← גיבוי אוטומטי לפני כל שמירה
│   └── 9294-export-old-format.xlsx         ← ייצוא לפורמט ישן (אם נדרש)
│
├── 5501/                                   ← יחידה אחרת
│   ├── 5501-data.xlsx
│   └── ...
```

**קונבנציות שמות:**
- `{unit}-data.xlsx` — קובץ העבודה המרכזי
- `{unit}-data.xlsx.backup-{timestamp}` — גיבוי (timestamp = ISO-like, ללא נקודתיים)
- `{unit}-export-old-format.xlsx` — ייצוא לפורמט הישן הידני (ראה סעיף 9)

## 4. פורמט Excel — ישן מול חדש מומלץ

### 4.1 ניתוח הפורמט הישן הקיים

על בסיס הקובץ `9294 - תחמושת אירן 2.cleaned.xlsx`:

**מבנה הגיליונות:**
| גיליון | תוכן |
|---------|-------|
| `ריכוז גדודי` | סיכום ברמת גדוד: כל סוגי תחמושת × סוללות עם עמודות מצב, שצ"ל ירי, עדכון |
| `תחמושת סוללה א` | פירוט סוללה: שורות = סוגי תחמושת, עמודות = קנים + מיקומי אחסון (בונקר/רמסע/ד') |
| `תמחושת סוללה ב` | כנ"ל לסוללה ב |
| `יחמור ג` | כנ"ל לסוללה ג |
| `עותק של ריכוז גדודי` | ריכוז עם עדכוני ספירה נוספים (גרסאות זמן) |

**מבנה גיליון סוללה (פירוט):**

```
    שורה 0 (כותרות): תחמושת | 1א | 1ב | 1ג | 2א | 2ב | 2ג | מפיק | ד1 | ד2 | ... | רמסע 1 | רמסע 2 | ... | סה"כ קנים | סה"כ בונקר | סה"כ
    
    שורות 1-9:   פגזים (M107, M107A3, M795, M485, M825, דמוי, M150, M401, M999)
    שורות 10-18: מרעומים (582, 739, 137, 762, 147, RTZ148, סרבי, 984, 557)
    שורות 19:    שורה ריקה
    שורות 20+:   חנ"ה — M7 אחוד (לוטים!), M7 פרוד (לוטים!), M9 (לוטים!), M10, M231, תחלים
    שורות 50+:   התאמות, פערים, חולץ, בלי טבעת
```

**שורת חנ"ה (charges) — פירוט לוט:**
- לפי הפורמט הנוכחי, חנ"ה **לא** מסוכמת בשורה אחת — כל לוט הוא שורה נפרדת:
  - `M7 אחוד 1-94/99`, `M7 אחוד 2-24`, `M7 אחוד 01-24` וכו'
  - `M9 2-02`, `M9 3-09`, `M9 1-07` וכו'
- הסיכום בריכוז הגדודי הוא **סך כל הלוטים** לכל סוג.

**עמודות:**
- **קנים** (guns): מזוהים ב-ID: `1א`, `2ב`, `813375` וכו' — אלה מזהי תותח/כלי
- **בונקר/אחסנה**: `ד1`, `ד2`, `רמסע XXX` — מיקומי אחסון
- **מפיק** — כלי/מכשיר מפיק
- **סה"כ קנים**, **סה"כ בונקר**, **סה"כ** — עמודות סיכום עם נוסחאות

**בעיות בפורמט הישן:**
1. ❌ **אין מבנה אחיד** — כל סוללה עם מספר עמודות שונה, שמות עמודות שונים
2. ❌ **לוטים לא מובנים** — שם הלוט הוא טקסט חופשי בתא A, לפעמים עם רווחים מיותרים
3. ❌ **#REF! errors** — נוסחאות שבורות ברקע הגדודי (כנראה עמודה ד' של "סוללה ד - זעורה" נמחקה)
4. ❌ **?** בערכים — `"197?"`, `"31?"`, `"0?"` — ערכים לא ודאיים, לא parseable כמספר
5. ❌ **תאים ריקים כ-"0"** — לא ברור אם ריק = 0 או ריק = לא ידוע
6. ❌ **עמודות כפולות** — עמודות Y-AA בגיליון סוללה מכילות ריכוז כפול (redundant)
7. ❌ **שמות לא עקביים** — `תמחושת סוללה ב` (שגיאת כתיב: תמחושת במקום תחמושת), ` יחמור ג` (רווח מוביל)
8. ❌ **merged cells** — משבשים עיבוד אוטומטי
9. ❌ **אין הפרדה בין סוגים** — פגזים, מרעומים, חנ"ה, תחלים — הכל באותו גיליון ברצף
10. ❌ **אין metadata** — לא ידוע מי עדכן, מתי, ומאיפה

### 4.2 הפורמט החדש המומלץ

#### עקרונות:
- **קריא לאדם ב-Excel** — כל משתמש שפותח ב-Drive רואה טבלה ברורה
- **Parseable ע"י מכונה** — עמודות קבועות, שמות אחידים, בלי merged cells
- **תאימות לאחור** — כולל את כל המידע שיש בפורמט הישן + עוד
- **לוטים מובנים** — מידע מלא על כל לוט
- **מטא-דאטא** — גיליון metadata נסתר עם version, timestamps, creator

#### גיליון `_metadata` (מוסתר/ראשון):

| שדה | ערך |
|------|------|
| `app_version` | `1.0` |
| `format_version` | `2` |
| `created_by_app` | `true` |
| `unit_number` | `9294` |
| `unit_name` | `גדוד 9294` |
| `unit_level` | `battalion` |
| `sub_units` | `סוללה א, סוללה ב, סוללה ג` |
| `created_at` | `2026-03-10T14:30:00Z` |
| `last_modified_at` | `2026-03-10T16:45:00Z` |
| `last_modified_by` | `user@example.com` |
| `data_revision` | `17` (מונה שמירות — לזיהוי שינויים) |

#### גיליון `config` — הגדרת יחידה:

| שדה | סוללה א | סוללה ב | סוללה ג |
|------|---------|---------|---------|
| `display_name` | סוללה א | סוללה ב | סוללה ג |
| `code_name` | כפר גלעדי | גומא | יחמור |
| `guns` | 1א, 1ב, 1ג, 2א, 2ב, 2ג | 1א, 1ג, 2א, 2ב, 2ג | 813375, 814553, ... |
| `storage_locations` | ד1, ד2, רמסע 1, ... | ד1, רמסע 326, ... | ד1, ד2, רמסע 1786, ... |
| `extra_locations` | מפיק, 621, 140 | מפיק | מפיק |

#### גיליון `סוללה א` (אחד לכל תת-יחידה):

**כותרות (שורה 1):**
| ammo_type | model | lot | 1א | 1ב | 1ג | 2א | 2ב | 2ג | מפיק | ... | סה"כ קנים | סה"כ אחסון | סה"כ |
|-----------|-------|-----|-----|-----|-----|-----|-----|-----|------|-----|-----------|-----------|------|

**שורות נתונים:**
| ammo_type | model | lot | 1א | 1ב | ... | סה"כ קנים | סה"כ אחסון | סה"כ |
|-----------|-------|-----|-----|-----|-----|-----------|-----------|------|
| shell | נפיץ M107 | | 29 | 16 | ... | 124 | 0 | 124 |
| shell | נפיץ M107A3 / סרבי | | 11 | 22 | ... | 36 | 0 | 36 |
| fuse | 582 | | 45 | 6 | ... | 132 | 24 | 156 |
| charge | M7 אחוד | 1-94/99 | 0 | 0 | ... | 0 | 0 | 0 |
| charge | M7 אחוד | 2-24 | 0 | 1 | ... | 1 | 0 | 1 |
| charge | M7 פרוד | 22/25 | 0 | 48 | ... | 121 | 0 | 121 |
| charge | M9 | 2-02 | 16 | 13 | ... | 61 | 100 | 161 |
| primer | תחלים | | 58 | 29 | ... | 265 | 201 | 466 |

**מה שונה מהפורמט הישן:**
1. ✅ **עמודת `ammo_type`** — קטגוריה ברורה (shell/fuse/charge/primer/bullet)
2. ✅ **עמודת `model`** — שם דגם אחיד
3. ✅ **עמודת `lot`** — מספר לוט כעמודה נפרדת (לא כחלק משם השורה)
4. ✅ **עמודות סיכום עם נוסחאות** — SUM אוטומטי
5. ✅ **אין merged cells** — כל תא עצמאי
6. ✅ **כותרות באנגלית ב-Row 0** (לזיהוי מכונה) + **כותרות עברית ב-Row 1** (לקריאה)
7. ✅ **ערכים מספריים בלבד** — ריק = 0, אין "?" או טקסט בתאי כמות

#### גיליון `ריכוז` — סיכום גדודי:

| ammo_type | model | סוללה א | סוללה ב | סוללה ג | סה"כ גדוד |
|-----------|-------|---------|---------|---------|-----------|
| shell | נפיץ M107 | 124 | 181 | 224 | 529 |
| shell | נפיץ M795 | 137 | 209 | 126 | 472 |
| ... | ... | ... | ... | ... | ... |

- כל תא סוללה = נוסחא `=SUM(...)` שמצביעה לגיליון הסוללה.
- מתעדכן אוטומטית כשהאפליקציה שומרת.

## 5. זרימת משתמש — יחידה חדשה (Clean Sheet)

```
    ┌──────────────────┐
    │  כניסה לאפליקציה │
    └────────┬─────────┘
             │
    ┌────────▼─────────┐
    │  התחברות Google   │  ← Scope: drive.file בלבד
    └────────┬─────────┘
             │
    ┌────────▼──────────────────┐
    │  בחירת תיקייה:            │
    │  [צור חדשה] [הדבק קישור]  │
    └────────┬──────────────────┘
             │
    ┌────────▼─────────┐
    │  בחירת מצב:       │
    │  [יחידה חדשה]     │  ← זה
    │  [עדכון יחידה]    │
    └────────┬─────────┘
             │
    ┌────────▼──────────────────────────┐
    │  הגדרת יחידה:                      │
    │  • מספר יחידה: [9294]              │
    │  • שם מדווח: [שמוליק]              │
    │  • רמת יחידה: [גדוד ▼]            │
    │  • תתי-יחידות:                     │
    │    סוללה א [כפר גלעדי] קנים: [6]   │
    │    סוללה ב [גומא] קנים: [6]        │
    │    + הוסף תת-יחידה                  │
    │  • מיקומי אחסון לכל תת-יחידה       │
    └────────┬──────────────────────────┘
             │
    ┌────────▼─────────────────────────┐
    │  נוצרים:                          │
    │  • תיקייה 9294/ ב-Drive           │
    │  • 9294-data.xlsx עם:              │
    │    - גיליון _metadata              │
    │    - גיליון config                  │
    │    - גיליון לכל סוללה (ריק)         │
    │    - גיליון ריכוז                   │
    └────────┬─────────────────────────┘
             │
    ┌────────▼─────────┐
    │  מסך ספירה        │  ← הקלדת כמויות
    │  (AmmoForm)       │
    └────────┬─────────┘
             │
    ┌────────▼─────────┐
    │  שמירה ל-Drive    │  ← כותב ל-xlsx, מעדכן revision
    └──────────────────┘
```

## 7. זרימת עריכה ושמירה — Race Conditions

### 7.1 הבעיה
שני אנשים עורכים את אותה יחידה במקביל:
- משתמש א' פותח את 9294, רואה revision 15
- משתמש ב' פותח את 9294, רואה revision 15
- משתמש א' שומר → revision 16
- משתמש ב' שומר → **דורס** את revision 16 ← אובדן נתונים!

### 7.2 הפתרון — Optimistic Locking

```
    ┌──────────────────────────────────────────┐
    │  לפני כל שמירה:                           │
    │                                           │
    │  1. קרא revision נוכחי מ-Drive             │
    │  2. השווה ל-revision שנטען בהתחלה          │
    │                                           │
    │  אם שווה → שמור + הגדל revision            │
    │                                           │
    │  אם שונה → ⚠️ "הנתונים שונו ע"י מישהו    │
    │     אחר מאז שפתחת. "                      │
    │                                           │
    │     [טען מחדש ואבד שינויים]                │
    │     [שמור בכל זאת (דרוס)]                  │
    │     [שמור כעותק נפרד]                      │
    │     [ביטול]                                │
    └──────────────────────────────────────────┘
```

### 7.3 מנגנון ה-Revision

- שדה `data_revision` בגיליון `_metadata` — מספר שלם שעולה ב-1 בכל שמירה.
- שדה `last_modified_at` + `last_modified_by` — מראים מי שמר אחרון.
- בטעינה: האפליקציה שומרת את ה-revision המקומי.
- לפני שמירה: קוראת revision מ-Drive ומשווה.

### 7.4 שמירה אוטומטית

- **אין שמירה אוטומטית ל-Drive** — רק בלחיצה מפורשת על "שמור".
- הסיבה: שמירה תכופה מדי → יותר סיכוי ל-conflict + בזבוז bandwidth מהשטח.
- **כפתור "שמור" בולט** עם מילים ברורות: `"שמור ל-Drive"`.

### 7.5 מצב "New" לעומת "Update"

| | יחידה חדשה (New) | עדכון יחידה (Update) |
|---|---|---|
| **שמירה** | יוצר קובץ חדש, revision=1 | בודק revision לפני שמירה |
| **Conflict** | לא אפשרי (קובץ חדש) | אפשרי — מנגנון optimistic locking |
| **גיבוי** | לא נדרש (אין מה לגבות) | גיבוי אוטומטי לפני כל שמירה |

## 9. ייצוא לפורמט ישן

### 9.1 מתי מוצג

כפתור "ייצא לפורמט ישן" מוצג **רק** כאשר:
- הנתונים נטענו ממיגרציה מקובץ ישן, **או**
- המשתמש ביקש זאת במפורש מהגדרות.

**לא מוצג** כאשר:
- הנתונים נוצרו מאפס באפליקציה (clean sheet) — אין סיבה לייצא לפורמט ישן.

### 9.2 מה נוצר

קובץ `{unit}-export-old-format.xlsx` עם:
- גיליון `ריכוז גדודי` — בפורמט זהה לישן (סוללות כעמודות, תחמושת כשורות)
- גיליון לכל סוללה — בפורמט זהה (קנים + אחסון כעמודות, חנ"ה עם שורות לוט)
- **עמודות סיכום עם נוסחאות** (SUM)

### 9.3 מגבלות הייצוא לפורמט ישן

- ⚠️ Merged cells לא נוצרים — הפורמט ניתן לקריאה אך ייראה מעט שונה.
- ⚠️ עמודות ריכוז כפולות (עותק של ריכוז) לא נוצרות — רק ריכוז אחד.
- ⚠️ ערכי "?" לא קיימים — כל ערך הוא מספרי (0 או חיובי).

## 10. שיקולי מיגרציה ואימוץ

### 10.1 יחידה חדשה שמתחילה עם האפליקציה

**במה נדרש:**
1. מוגדרים משתמשי test ב-Google OAuth (עד 100 כרגע).
2. משתמש מתחבר, האפליקציה יוצרת תיקייה.
3. מגדיר יחידה — שמות סוללות, מספר קנים, מיקומי אחסון.
4. מתחיל לספור.

**מה חשוב:**
- Template של סוגי תחמושת — מוטמע באפליקציה כברירת מחדל, ניתן להתאמה.
- אם unit_level = battery, אין תתי-יחידות — מסך ספירה ישיר.
- אם unit_level = battalion, יש tabs לכל סוללה + ריכוז.

### 10.2 יחידה קיימת שעוברת מניהול ידני

**שלבי מיגרציה:**

```
    שלב 1: המשתמש מעלה/מצביע את הקובץ הישן
           ↓
    שלב 2: האפליקציה מנתחת ומציגה תצוגה מקדימה
           ↓
    שלב 3: המשתמש מאשר, נוצר קובץ חדש בפורמט המומלץ
           ↓
    שלב 4: מעכשיו — האפליקציה עובדת עם הקובץ החדש
           ↓
    שלב 5: אם צריך — ייצוא לפורמט ישן להשוואה/הפצה
```

**חששות למשתמש:**
- "מה קורה עם הקובץ הישן?" → **לא ייערך, לא יימחק**.
- "אם יש טעות?" → **גיבוי לפני כל שמירה + שחזור**.
- "איך אני שולח דוח לא דרך האפליקציה?" → **ייצוא לפורמט ישן** או **הורדת Excel**.
- "שני אנשים עורכים?" → **זיהוי conflict + הודעה ברורה**.

### 10.3 תאימות לאחור — מה נשמר

| מה יש בפורמט הישן | מה יש בפורמט החדש | סטטוס |
|---|---|---|
| סוגי תחמושת (פגזים, מרעומים, חנ"ה, תחלים) | ✅ כל הסוגים + קטגוריה מפורשת | שופר |
| פירוט לוטים לחנ"ה | ✅ עמודת lot מובנית | שופר |
| מיקום: קנים + בונקר + רמסעים | ✅ הכל כעמודות | שווה |
| עמודות סיכום (SUM) | ✅ נוסחאות SUM | שווה |
| ריכוז גדודי | ✅ גיליון ריכוז עם נוסחאות | שווה |
| Merged cells | ❌ מבוטלים | שינוי מכוון |
| עותק כפול של ריכוז | ❌ לא נוצר | שינוי מכוון |
| שורות "התאמות", "פערים", "חולץ" | ✅ נשמרים כשורות metadata | שווה |
| ❌ אין מעקב מי/מתי | ✅ metadata מלא | חדש |
| ❌ אין גיבוי | ✅ גיבוי אוטומטי | חדש |
| ❌ אין זיהוי conflict | ✅ optimistic locking | חדש |

### 10.4 טמפלט סוגי תחמושת

- ברירת מחדל: רשימת סוגים מובנית באפליקציה (הרשימה הנוכחית).
- ניתן לעדכן דרך קובץ `ammunition-types.xlsx` בתיקייה הראשית ב-Drive.
- אם הקובץ קיים — האפליקציה קוראת ממנו. אם לא — משתמשת בברירת מחדל.
- המשתמש יכול לערוך את הקובץ ב-Drive ולהוסיף/למחוק סוגים.
