# הקשר ארכיטקטוני והמלצות פיתוח — אפליקציית ספירת מלאי תחמושת (PWA)

## מטרת המסמך
המסמך הזה מיועד ל-AI agent בתוך VSCode. המטרה שלו היא **לא לכפות מימוש אחד קשיח**, אלא לתת הקשר ארכיטקטוני, עקרונות תכנון, אילוצים, והמלצות מודולריות לבניית המערכת.

הדרישות כאן הן **ראשוניות ויכולות להשתנות**. לכן יש להתייחס אליהן כהכוונה, לא כמפרט סופי.

---

## תיאור כללי של המוצר
מדובר באפליקציית ווב מסוג **PWA** לניהול וספירת מלאי תחמושת ביחידה צבאית.

היעד הוא מערכת פשוטה, מהירה, נוחה לשימוש בשטח, בעברית וב-RTL, שעובדת בדפדפן ויכולה להיות מותקנת כאפליקציה באנדרואיד.

### כיוון מוצרי מרכזי
האפליקציה היא בעיקר:
- UI נוח יותר מאקסל / Google Sheets
- כלי להזנת נתונים מסודרת לפי היררכיה צבאית
- כלי לחישוב אוטומטי של סיכומים וטוטאלים
- כלי לייצוא דוחות ל-Excel ול-PDF
- כלי לשמירת טיוטות והמשך עבודה

### נקודת מוצא חשובה
לפי האפיון הנוכחי, הכיוון המועדף כרגע הוא **local-first**:
- הנתונים נשמרים מקומית במכשיר
- אין חובה לשרת מרכזי
- אין backend חובה
- אין בסיס נתונים מרכזי
- האפליקציה צריכה לעבוד גם בלי תשתית שרתים

עם זאת, כדאי לתכנן אותה **באופן מודולרי** כך שאפשר יהיה בעתיד:
- להוסיף סנכרון
- להוסיף שכבת אחסון אחרת
- להוסיף import/export בין מכשירים
- להוסיף נעילה או merge אם יעברו לעבודה משותפת

## עקרונות ארכיטקטורה מומלצים

### 1. Frontend-only כברירת מחדל
ההמלצה הראשונית היא לממש את האפליקציה כ-**client-side app בלבד**:
- React + TypeScript + Vite הם כיוון טוב
- אין צורך ראשוני בשרת אפליקטיבי
- אין צורך ראשוני ב-API משלך

הסיבה:
- פחות מורכבות
- קל להרים מהר
- מתאים לשימוש אד-הוק
- מתאים ל-PWA
- מתאים לשמירה מקומית

### 2. Local-first data model
האפליקציה צריכה להניח שהעותק העיקרי של הנתונים נשמר מקומית במכשיר.

המלצה:
- IndexedDB כאחסון ראשי
- LocalStorage רק לדברים קטנים כמו preferences

לדוגמה:
- שם מדווח אחרון
- רמת משתמש אחרונה
- יחידה אחרונה שנבחרה
- הגדרות UI בסיסיות

### 3. מודולריות של שכבת האחסון
למרות שהמימוש הראשוני הוא local-only, כדאי לבנות abstraction ברור לשכבת האחסון.

מומלץ לא לקשור את כל האפליקציה ישירות ל-IndexedDB.

עדיף להגדיר ממשק כמו:
- saveDraft
- loadDraft
- listDrafts
- exportReport
- importReport
- clearCurrentReport

כך אפשר בעתיד להחליף backend storage בלי לשנות את שאר האפליקציה.

### 4. Domain-driven structure
מומלץ להפריד בין:
- UI components
- domain model
- state management
- persistence
- report generation
- import/export
- configuration/catalog

לא מומלץ לשים הכל בתוך קומפוננטות React.

### 5. Catalog-first, data-second
סוגי החימוש באפיון כרגע ידועים מראש, אבל יכולים להשתנות.

לכן כדאי לבנות את המערכת כך ש:
- קטלוג החימושים יוגדר בקונפיגורציה / data definitions
- טפסי ההזנה ייגזרו ממנו
- הדוחות ייגזרו ממנו
- ערכים שלא הוזנו יוצגו כ-0 על בסיס הקטלוג, ולא רק על בסיס מה שהמשתמש הזין

זה חשוב במיוחד כי יש דרישה מפורשת:
**חימוש שלא נבחר חייב להופיע בדוח עם ערך 0.**

## קטלוג חימושים — הקשר תכנוני

### סוגי ישויות
לפי האפיון כרגע יש לפחות 5 קבוצות:
1. פגז
2. חנ"ה
3. מרעום
4. תחלים
5. קליעית

### המלצה חשובה
לא לקודד את הטפסים ידנית לכל פריט, אלא להחזיק **catalog/schema configuration** שמתאר:
- group type
- display label
- allowed models
- extra fields per row
- validation rules
- whether quantity only / quantity + model / quantity + model + serial

למשל:
- פגז = model + quantity
- חנ"ה = model + serial + quantity
- מרעום = model + quantity
- תחלים = quantity only
- קליעית = model + quantity

כך ה-UI נהיה data-driven וקל להרחבה.

## דרישת ה-0 בדוחות
דרישה מהותית:
כל סוג חימוש שלא נבחר עדיין צריך להופיע בדוח עם כמות 0.

השלכה ארכיטקטונית:
לא לבנות דוח רק ממה שקיים ב-input rows.

מומלץ שהדוח ייווצר כך:
1. טוענים catalog מלא
2. טוענים user entries
3. עושים normalization
4. מייצרים output מלא עם כל האפשרויות

כלומר, חסר נתון = 0, לא “לא קיים”.

## טעינת קבצים קיימים
האפיון כרגע מדבר על:
- פלגה יכולה לטעון קבצי צוות קיימים
- סוללה יכולה לטעון קבצי פלגה קיימים
- עדכון אוטומטי

### המלצה
כרגע אין הכרח להחליט על sync מלא. מספיק לתת ל-AI agent להבין שצריך לבנות import pipeline מודולרי.

כיוון מוצע:
- כל דוח/טיוטה יהיה בר ייצוא כקובץ JSON פנימי
- אפשר יהיה לטעון JSON של צוות לתוך פלגה
- אפשר יהיה לטעון JSON של פלגה לתוך סוללה
- ה-merge logic צריך להיות מופרד משכבת ה-UI

### עקרון חשוב
ה-agent לא חייב להתחייב עכשיו למנגנון sync יחיד.
עדיף שיבנה abstractions ל:
- importUnitReport
- mergeIntoParentReport
- validateImportedScope
- detectOverwriteOrDuplicate

כך אפשר בעתיד לבחור אם הייבוא מגיע מ:
- קובץ
- Drive
- storage אחר

למימוש ראשוני אפשר:

- לאפשר import

ואם סביבת היעד תומכת, אפשר אחר כך להרחיב לשמירה ישירה יותר.

## אבטחה ורגישות מידע
האפיון הנוכחי מזכיר שמירה מקומית, אבל ברקע יש רגישות אפשרית למידע מבצעי.

לכן כהמלצה ארכיטקטונית כללית:
- לא להניח שהנתונים חסרי רגישות
- לצמצם תלות בשירותים חיצוניים
- לא לשמור מידע אמיתי בתוך קוד המקור
- להפריד בין catalog/configuration לבין data operational
- לשקול export/import שניתן לשליטה תפעולית

עם זאת, ה-agent לא צריך להיכנס כרגע למנגנוני security enterprise אלא רק לתכנן קוד שלא נועל אתכם לפתרון מסוים.

## המלצה מעודכנת ביחס לאפיון הנוכחי
ביחס למה שנדון קודם על Drive/sync/lock:

### אם הולכים לפי האפיון הנוכחי כלשונו
ההמלצה הראשונית היא **לא להתחיל מ-Google Drive sync** ולא מנעילות cross-device.

למה?
כי האפיון הנוכחי מדגיש:
- שמירה מקומית במכשיר
- PWA פשוטה
- שימוש מהיר בשטח
- בלי תשתית שרתים

לכן לשלב ראשון עדיף שה-agent יבנה:
- local-first architecture
- drafts מקומיים
- import/export לקבצים
- merge בסיסי בין דוחות אם צריך

### מה כן כדאי להכין מראש
כדי לא לחסום עתידית sync, אפשר להמליץ ל-agent:
- להגדיר storage interface מופשט
- להגדיר report serialization format יציב
- להפריד import/export logic
- להפריד locking/sync services גם אם הם עדיין ממומשים כ-noop או local-only

כלומר: **להתכונן להרחבה, אבל לא לממש כרגע תשתית סנכרון חיצונית אם לא צריך.**

## מה כן חשוב שה-agent יפנים
1. זה PWA פשוט, usable בשטח, בעברית, RTL.
2. כרגע הכיוון העיקרי הוא local-first ולא server-first.
3. הדומיין בנוי סביב היררכיה של צוות / פלגה / סוללה.
4. יש קטלוג חימושים מוגדר, אבל הוא צריך להיות configurable.
5. חימוש שלא הוזן חייב להופיע בדוחות עם 0.
6. צריך ייצוא ל-Excel ול-PDF.
7. כדאי לתכנן שכבות מודולריות כדי לאפשר הרחבות עתידיות.
8. import/export ו-merge חשובים יותר כרגע מ-sync מורכב.

## בקשת יישום מומלצת ל-agent
אם ה-agent צריך לבחור כיוון מימוש, עדיף שיתחיל כך:

1. יבנה domain model נקי וגמיש
2. יבנה catalog-driven forms
3. יממש local storage abstraction
4. יממש drafts ב-IndexedDB
5. יבנה report normalization עם אפסים כברירת מחדל
6. יבנה export ל-JSON + Excel + PDF
7. יוסיף PWA installability
8. ישאיר hooks/extensions עתידיים ל-sync/import/merge

## סיכום
המערכת כרגע לא צריכה להיבנות כמו מערכת ארגונית כבדה, אלא כמו:

**אפליקציית PWA מקומית, מודולרית, נוחה לשטח, שמחליפה עבודה ידנית באקסל, ושומרת על אפשרות התרחבות עתידית.**

הסדר הנכון הוא:
- קודם domain נכון
- אחר כך טפסים ודוחות
- אחר כך שמירה מקומית טובה
- אחר כך import/export
- ורק בהמשך, אם בכלל, סנכרון חיצוני ונעילות בין מכשירים.
